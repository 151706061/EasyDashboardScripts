EasyDashboardScripts

Wouldn't it be nice if, on any computer, you could do this to run a dashboard for your favorite software project?

 

  ctest -S FavoriteProject.cmake

 

And wouldn't it be even nicer if the contents of FavoriteProject.cmake looked as simple as this?

 

  SET(dir "${CTEST_SCRIPT_DIRECTORY}/../EasyDashboardScripts")

  INCLUDE("${dir}/EasyDashboard.cmake")

 

And wouldn't it just be totally sweet if you could even modify how the dashboard builds by adding some simple parameters to the ctest call like this?

 

  ctest -S FavoriteProject.cmake,Debug-Shared-WrapPython

 

It would be nice. Actually, it is nice. Sweet! It's even entirely possible with the scripts found in EasyDashboardScripts.

Easy Setup

Here's the bare bones bootstrap sequence to use EasyDashboard on your computer:

  • Create a Dashboards directory
  • Put EasyDashboardScripts under Dashboards
  • Create "Dashboards/My Tests" (with a space)
  • Put your source trees under "My Tests" (with cvs, svn or whatever…)
  • cd into EasyDashboardScripts and run a dashboard with "ctest –S ProjectName.cmake"

 

Scripts are included in EasyDashboardScripts to run dashboards for the following projects without any additional work:

  • CableSwig
  • CMake
  • ParaView
  • KWWidgets
  • VTK

 

Additional work is only required if you want to override the default settings that EasyDashboard uses. If you want to learn more about how it works or customize it for your machine, then read on…

Goals and Conventions

Let's start by stating some goals and agreeing on some conventions to adhere to on any machine that runs dashboards.

 

Goals

Write a single dashboard script for a project that will drive a dashboard for that project on *any* machine with *any* development environment.

Set up a new machine to run dashboards with almost zero effort. If you have CMake 2.4 or later installed and a development/build environment, you should be able to run a dashboard with only a few *minutes* worth of prep work. Anybody anywhere who can build a project with CMake and a native build tool should be able to run a dashboard for that project.

Alter a dashboard script slightly (or even radically) by passing arguments in with the script to control such things as build type/configuration, development environment, wrapping languages and initial cache settings.

Use "new style" CTest 2.4 dashboard scripts to eliminate double configures and reduce dashboard run times.

 

Conventions

User name: run dashboards as a non-privileged user named "dashboard" – share the password for this user with other people around you who may need to logon to your machine when you're not around to investigate a test failure that only occurs on your machine.

Directory structure: have a "Dashboards" directory, typically in the dashboard user's home directory, or at "C:\Dashboards" on a Windows machine. Have standard subdirectories as follows:

  Dashboards

    EasyDashboardScripts

    Logs

    My Tests

    Support

The project source trees should reside in the "My Tests" subdirectory. For example,

  Dashboards

    My Tests

      CMake

      SourceTree3

      SourceTree4

      VTK

The project binary/build trees should reside as siblings of the source tree in the "My Tests" subdirectory.

  Dashboards

    My Tests

      FavoriteProject

      FavoriteProject Win32-vs8-Debug

      FavoriteProject Win32-vs8-Release

      SourceTree3

      SourceTree3 Win32-vs8-Debug

      SourceTree3 Win32-nmake8-MinSizeRel

      SourceTree4

      SourceTree4 Win32-borland-Debug

      SourceTree4 Win32-cygwin-MinSizeRel

The "My Tests" directory name should have a space in it.

The names of the binary trees are the names of the source trees appended with a space and some configuration parameter values.

Project script location: place project scripts in any subdirectory of Dashboards, such that "${CTEST_SCRIPT_DIRECTORY}/.." resolves to the Dashboards directory. Following this convention allows project scripts to use:

  SET(dir "${CTEST_SCRIPT_DIRECTORY}/../EasyDashboardScripts")

  INCLUDE("${dir}/EasyDashboard.cmake")

Following the project script location convention also allows separate directories for separate groups of project scripts if necessary (perhaps originating from different CVS or SVN repositories), yet they can all use the same technique for obtaining the EasyDashboard functionality.

A directory structure like this may ensue:

  Dashboards

    EasyDashboardScripts

    MyDashboardScripts

    ProjectDashboardScripts

Do not use the prefix "ED_" for CMake variables in project scripts. ED_ names are reserved for use by EasyDashboard.cmake and its associated scripts.

Argument composition: alphanumeric and underscores only. *No* spaces, quotes, semi-colons, colons, commas, dashes, braces, other punctuation or "special characters" are allowed in script arguments that control dashboard run parameters. Project specific arguments should be alphanumerics that begin with a project specific prefix followed by an underscore, similar to typical CMakeLists variable naming conventions.

Argument uniqueness: all possible arguments should be completely orthogonal to all other possible arguments with respect to regular expression matching in CMake scripting. Ideally, no argument should match any other argument when comparing each pairwise with every other on either side of a CMake MATCHES comparison. See the Arguments section for more discussion on this point. (Note: Arguments section still to be written…)

Attach any additional custom cmake scripts used when processing your main project script (an override script, for example) as dashboard notes using this construct as the very first line of the additional script:

 

  SET(ED_notes ${ED_notes} "${CMAKE_CURRENT_LIST_FILE}")
